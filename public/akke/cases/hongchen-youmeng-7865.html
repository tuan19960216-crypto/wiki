ACT 03 · 沉默的 9 天 2026-05-09 21:41 → 2026-05-18 22:08 · 队列里的 217 小时

高意向评论入库后默认进入反向评论（RC）队列，等待运营按当日额度批量审核 + 投放。本案的 RC1 一直到 5-18 22:08:50 CST 才发出 — 距客户原始评论时间 217 小时（9 天 0.5 小时）。

为什么会这么慢？
① 视频被识别为 topic_status=rejected，source_account_id=null — 不在每日号源轮询里，只能等运营手动用 profile 扫描兜底捡到（5-15 / 5-16 之间）。
② RC 投放靠运营每日选 10-20 条批量审过后入设备，单条排队周期 1-3 天属正常。
③ 结果："评分准"但"接住慢" — funnel 上游正确，但 SLA 远未达到能"接住"高意向客户的水准。

ACT 05 · 第二次回应（RC2）· 含 bug 2026-05-20　17:18:27 CST · 距 RC1 + 43 小时

RC2 出了两个生成 bug — 地域抖动 + 数字矛盾

因为客户对 RC1 没回，2 天后 LLM 又生成了第二条 RC（见 #2）作为第二轮触达。这条 RC 没复用 RC1 的关键事实，结果出现两处真实漏洞，让我们这一行的专业感打了对折。

🐞 Bug #1 · 地域写错为「湖南」
原文："我们自建房整装做过不少湖南类似户型"
客户 IP 属地是安徽（comments.ip_location = "安徽"）。LLM 在生成 RC2 时没读 RC1 的 context，也没看 ip_location，凭空抛了"湖南"。
影响：如果客户点开看 RC1 + RC2 会发现一会儿不绑地域、一会儿绑错地域 — 直接破坏"懂你家这边"的人设。
🐞 Bug #2 · 下沉数字与 RC1 矛盾
RC1："下沉 15-20 公分给排水管走"
RC2："建议下沉 25-30 cm 留好排水坡度"
两条 RC 给的施工数字差了一档（15-20 vs 25-30）。LLM 没有"记住"自己上一条说过什么。
影响：客户如果对照看两条，会怀疑「这两个回复是不是同一个人 / 同一家公司」— RC2 的可信度直接被 RC1 反着拖下来。
RC 二刷规则（应当落入 prompt）
① 地域必须复用 IP 属地，不许凭空换省 — 应在 system prompt 里硬约束 ip_location。
② 技术数字必须复用第一条（或显式给出"修正前后"），不允许独立生成。
③ RC2 应感知 RC1 内容（按 douyin_user_id + video_id 拉最近一条已发 RC 作为 context）。

复盘 · 这条线索教会我们的

6 个必须修的系统问题

1
SLA 是一切。217 小时后的"高意向"回应 ≈ 不存在。当前 funnel 的 P95 时效不应超过 24h，知识型评论应当走专用加速队列，3h 内有效。
2
knowledge 型评论 = ROI 最高的一类。客户在问"做不做下沉 / 多少 cm" — 这是他正在做选择的实时窗口。错过窗口后即使文案完美也救不回，因为他已经问到答案了。
3
RC 二刷必须感知第一条。本案 RC2 把地域写成湖南、把下沉数字从 15-20 改到 25-30 — 这是因为 LLM 没拿到 RC1 作为 context。修：generate_rc2 必须 inject 最近一条已发的 RC 文本。
4
IP 属地必须硬约束。RC1 没用「安徽」、RC2 凭空写「湖南」— 都是 LLM 自由发挥导致。修：prompt 里 ip_location 设为必须 verbatim 出现的字段，模型不准换省。
5
价格钩子要在 RC1 抛出。本案 RC1 只说"拉清单"，没把"全屋柜体 5 万起 / 自建房整装报价"挂出来。高意向 + 进行中 + 高 specificity 的客户，价格信息是 disclose 钩子，不是营销噪音。
6
DM 开场必须按是否有客户回复分支。"不客气~"假设了一个不存在的前置交互。修：compose_dm 先查 messages where role='user'，没有就用"零起点"模板，不许用感谢应答模板。
