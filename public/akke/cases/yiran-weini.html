复盘 · 这条线索教会我们的

5 个审批环节 + draft 状态的原则

1
从评论到触达有 5 个串行环节 · 每个都可能卡住。本案让"打不到客户"的真实原因显形：①cron 评分（这次滞后 33 小时）；②LLM 起草（已完成）；③运营审批（已 10+ 小时未通过）；④账号绑定（messaging_account_id NULL）；⑤发送队列入队。5 步串行 = 5 个独立故障源 · 任何一步卡住客户都收不到。监控面板需要分环节 SLA 告警，不能等到客户没回应才发现 DM 根本没发。
2
"在飞"案例和"还没起飞"案例要区分。空间向量初版 / 即将是个辣妹的"在飞"都是 DM 已发但客户没回；本案是DM 还没发。从客户视角，这两种情况完全不同：前者她至少看到了我们；后者她甚至不知道我们存在。案例库的诚实展示原则：journey-strip 必须清晰区分这两档（本案"DM 触达"用 now 黄色而不是 done）。
3
知识型评论文案要"直接回答 + 抛工艺约束"。客户问"能不能加高" — draft 文案的第 ② 段直接答"小空间柜体加高太关键了，顶上吊柜做满一寸都不浪费"（直接答），第 ④ 段抛"板材幅面和结构承重 / 层高得算准"（工艺约束）。知识型问句 = 客户在评估，所以答案要兼顾"可以做"和"但需要看条件" — 前者打消"不能做"的疑虑，后者建立"我懂这事的复杂性"的专业感。
4
审批环节的 SLA 应当对应评论 freshness。本案评论 5-18 晚 22:06 留 → 5-20 07:33 评分 → 现在还在等审批，从评论到草稿 33 小时，从草稿到审批 10+ 小时。客户当时刷视频时的"想问这事"心态早就过去了。审批 SLA 应当跟评论freshness等级挂钩：①active (24h 内) freshness 必须 <4h 审批；②stale (>72h) 可以 24h 内审批；③等待越久客户越凉。
5
messaging_account_id = NULL 是连续两条案例都出现的工程 bug · 必须修。"即将是个辣妹" + 本案"依然为你"连续两条案例都发现 conversation 创建时没绑发送账号。这是系统性问题，不是偶然。工程优先项：①补 conversation create 路径在所有分支都强制绑 messaging_account_id；②扫描历史 NULL 的 conversation 做回填脚本；③加 DB 约束 NOT NULL 让新数据无法插入空值。
