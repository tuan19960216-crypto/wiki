ACT 02 · 被看见 2026-05-19　12:11 CST · 评分 cron 命中

破冰文案 V3 · 四段式定型
四段式 ice_break 的标准结构 — 不结束在客气话，而是结束在问题上。本条 DM 用了一个有趣的策略：把客户那句模糊的"最终一步"主动解读为"油工阶段"，赌的是"如果蒙对了，她会觉得我懂；蒙不对，她会纠正我"。这是 cold opener 里的小赌注 — 没有自报市可用、又是知识号场景，需要靠一个"懂行的具体性"破冰。客户的"价格"钩在第三段被回答（区间报价 5 万），第二轮如果有机会，可改为"先单独报柜体单价 vs 全屋包"对比展开。

⚠ 这是本案最有教育意义的一幕
"queue.status = failed" ≠ "客户没收到"。

detect_send_outcome 用关键字匹配 XCTest source XML 里的 placeholder banner（"对方回复后才能发消息"）来判定投递成功。但抖音 iOS App 把这串文案渲染成 image 或 custom control，不在 accessibility tree 里 — XCTest 的 source() 拿不到。

历史复盘（2026-05-02 批量 14 条 warmup）：algorithm 标 8 条 silent_drop_suspected，肉眼审 screenshot 后全部看见蓝色 chat bubble + placeholder banner = 真投递。

因此本案的正确解读是："已点击发送 + 回执失读 → 真实投递概率 ≈ 80%"，剩下 20% 才是真 silent drop。下一步必须打开本机 /tmp/warmup_*_after_send.png 看截图，看到蓝 bubble 就批量改 messages.status='sent' + message_queue.status='sent'。
ACT 05 · 等首次回应 已等不到 24 小时

她还没回 · 而且我们其实不确定她有没有看到

系统记录 messages 表里仅有 #1 一条 AI 消息，没有 role=customer 行。这是 ice_break 阶段常见的中间态，但本案叠加了一层不确定性 — 我们没法肯定 DM 真的进了她的私信框。在确认投递状态之前，"她还没回"和"她根本没看到"这两个分支无法被区分开来。Akke 不会在没回应时主动追问，避免把 sec_uid 烧成陌生人 DM 黑名单；本案推荐先人工审 screenshot 修正 queue 状态，再让 followup_policy = standard 自然推进。

对话还在 ice_break 阶段 · queue 状态卡在 failed 待审。

复盘 · 这条线索教会我们的

5 个可复用的系统观察

1
错字不抑评分。"价格对少" → "价格多少"是键盘相邻位的笔误，LLM 评分（DeepSeek V4 Flash）做了 fuzzy 容错没被误导。这条对 prompt 调参的启示是：不要在 noise filter 里加"必须是规范中文"过滤，否则会丢失大量真实高意向。
2
临近午夜的询价是装修业的高意向时段。23:50 这种时间点留言，往往意味着客户白天工地跑完、晚上正在比价决策。Akke 的评分模型把"freshness · active"权重设得偏高，是为了不让深夜评论在第二天醒来时就被 cron 当 stale 跳过。
3
知识号场景下，省级气候话术 = 同城回退路径。没有 comments.city、号源又是"全国"，硬同城判定一定 false。这时"安徽湿气重 → 板材选防潮"这种省级气候 + 产品约束的组合是最稳的破冰锚 — 它证"我懂你的本地物理约束"，比泛报价更接得住产品咨询型评论。
4
queue.status='failed' 是 algorithm 的悲观判定 · 不是事实。detect_send_outcome 用 XCTest source XML 抓 placeholder banner，但抖音 iOS 把这串文案渲染成 image / custom control 让 accessibility tree 拿不到，历史 false-negative ≈ 80%。遇到 silent_drop_suspected 必须打开 screenshot 肉眼审，看见蓝 bubble = 真投递，批量改回 sent。把这条写成 runbook 比写在脑子里靠谱。
5
状态未澄清前不要追加 followup。如果 #1 其实送达，followup 会变叠加追问把客户推走；如果真没送达，问题在通道不在话术 — 追加文案也无效。诚实地把"待审"作为一个 stage 标出来，比假装已经触达然后追第二条更省客户的耐心。
